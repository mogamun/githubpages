# 투더제이 — 바이브코딩 치트키 15

## 개요
바이브 코딩 입문 Vol.3 (13분 20초). AI를 "잘 쓰는 것"이 아니라 "잘 조련하는" 15가지 방법론. PRD 작성부터 페르소나 부여, 기술 스택 고정, 컨텍스트 관리, 프롬프트 체이닝까지 순차적 가이드를 제공. 투더제이가 제안하는 5그룹 구조(시작 전 준비 → AI 소통 → 코드 관리 → 품질 향상 → 완성)로 바이브 코딩의 전체 라이프사이클을 커버한다.

## 핵심 내용

### 그룹 1: 시작 전 준비

**키 1 — PRD 먼저, 코드는 나중에**
AI에게 바로 "쇼핑몰 만들어 줘"라고 하면 AI가 자기 마음대로 만든다. 결과물이 내가 원하는 것과 완전히 달라진다. 해결법은 PRD(Product Requirements Document, 제품 요구사항 문서)를 먼저 작성하는 것. 프로젝트의 이름, 핵심 기능, 기술 스택, 페이지 구조를 AI에게 먼저 보여주면 AI가 설계도를 보고 집을 짓게 된다.

**키 3 — 역할을 부여하라 (페르소나)**
AI에게 페르소나를 부여하는 것. 프롬프트에 "너는 10년차 시니어 Next.js 개발자다"라고 한 줄만 추가해도 AI의 답변 퀄리티가 달라진다. 역할, 전문성, 코딩 스타일, 제약 조건까지 넣으면 시니어 개발자가 옆에 앉은 것과 같은 수준의 코드를 얻을 수 있다.

**키 8 — 기술 스택을 고정하라**
AI는 물어볼 때마다 다른 기술을 추천한다. 어제는 MongoDB, 오늘은 Supabase, 내일은 Firebase. 이러면 프로젝트가 엉망이 된다. 처음에 스택을 정하고 규칙 파일(CLAUDE.md, Cursor Rules)에 고정. 2026년 가볍게 시작하기 좋은 스택: Next.js + Supabase + Tailwind CSS.

### 그룹 2: AI와 소통하는 법

**키 4 — 컨텍스트를 사수하라**
AI는 건망증이 심하다. 대화가 길어지면 앞에서 한 말을 잊어버린다. 해결 방법 세 가지: (1) CLAUDE.md나 Cursor Rules 같은 규칙 파일 생성, (2) 대화가 길어지면 "지금까지 내용 요약해줘"라고 시키기, (3) 한 번에 너무 많은 걸 시키지 않기.

**키 5 — 한 번에 하나씩만 시켜라**
로그인 기능 만들면서 동시에 대시보드도 만들어 줘라고 하면 AI가 과부하에 걸린다. 하나도 제대로 못 만드는 경우가 발생. 원칙은 1프롬프트 = 1기능. 몇 개의 파일 정도 변경이라는 작게 시키고 확인하고 다음으로 넘어가는 것이 좋다.

**키 6 — 프롬프트 체이닝**
복잡한 기능은 한 번에 다 시키면 안 되고 쪼개서 순서대로 시킨다. 예를 들어 결제 기능이라면: 첫 번째 단계 = 데이터 모델 설계 → 두 번째 = API 생성 → 세 번째 = UI 구현. 이렇게 체이닝하면 복잡한 기능도 안전하게 구현된다.

**키 7 — 에러 메시지는 금이다**
에러가 나면 절대 당황하지 말고 에러 메시지를 통째로 복사해서 AI에게 던진다. "이 에러가 나왔는데 운영 환경은 이렇고 이런 걸 시도해 봤다"라고 세 가지 정도만 전달하면 AI가 30분짜리 디버깅을 30초 만에 해결해 준다.

### 그룹 3: 코드 관리법

**키 8 — 코드를 읽지 말고 구조를 읽어라**
초보자가 코드 한 줄 한 줄 읽을 필요가 없다. 대신 폴더 구조와 데이터 흐름만 이해하면 된다. 어떤 폴더가 무슨 역할인지, 데이터가 어디에서 어디로 흐르는지만 파악하면 프로젝트 전체를 조감할 수 있다.

**키 9 — AI 코딩 도구를 선택하라**
- **커서 (Cursor)**: 사용이 가장 쉽고 빠르게 접근 가능. 초보자에게 추천. 서브 에이전트 기능도 지원
- **클로드 코드 (Claude Code)**: 터미널에서 대규모 작업에 강함. 서버에서도 사용 가능
- **안티그래비티 (AntiGravity)**: 커서와 비슷한 인터페이스. 초심자도 쉽게 접근

자기 스타일에 맞는 도구를 고르는 게 중요. 세 가지를 다 사용해 보고 맞는 것을 선택하는 것도 좋은 방법.

### 그룹 4: 품질 높이기

**키 10 — 테스트는 AI에게 시켜라**
"이 함수에 대한 테스트 코드를 작성해 줘" 한 마디면 된다. 작성은 AI가, 판단은 내가. 무엇을 테스트할지만 내가 정하면 된다. AI가 결과물을 파악하면서 코드를 잡아가는 과정이 된다.

**키 12 — UI는 레퍼런스로 설명하라**
"예쁜 대시보드 만들어 줘"는 굉장히 모호한 프롬프트. 대신 원하는 디자인 스크린샷을 첨부하고 "이 레이아웃 참고해서 만들어 줘"라고 하자. 스크린샷 한 장이 100줄짜리 설명을 대체한다. 또한 "예쁜", "심플", "모던", "고급", "미니멀" 등의 형용사 조합을 사용하면 AI가 어떻게 디자인하는지 감이 잡힌다.

**키 13 — AI를 의심하라**
AI는 때로 자신감 넘치게 거짓말을 한다. 없는 라이브러리를 추천하고, 되지도 않는 코드를 완성이라고 하고, 여전히 에러가 나고 있는데 에러를 수정했다고 한다. 항상 결과물을 검증하고 대안을 탐색하는 습관을 길러야 한다.

### 그룹 5: 완성하기

**키 11 — 배포를 미루지 말라**
"완벽하게 만들고 배포하겠다"고 생각하면 영원히 배포하지 못하고 나 혼자만의 놀이로 끝날 수 있다. 첫날부터 Vercel이나 AWS에서 배포. 실제 URL이 생기면 동기 부여가 완전히 달라진다.

**키 14 — 리팩토링은 AI의 특기**
기능 3개를 만들 때마다 AI에게 "코드 정리해 줘"라고 말해보자. 중복 제거, 함수 분리, 네이밍 개선 등의 코드 정리가 자동으로 된다. 기술 부채가 쌓이기 전에 자동 청소.

**키 15 — 문서화는 미래의 나를 위한 선물**
3개월 뒤에 이 프로젝트를 다시 열면 자기가 뭘 만들었는지 기억이 안 난다. 하지만 README만 잘 써두면 3개월 뒤에 나에게 감사하게 된다. 개발 현황과 사용 방법 등을 README에 정리해 달라고 AI에게 요청.
